BLUF: Your planning doc covers feature decomposition well but underdefines the shared architectural components that parallel builders will depend on. Below are the cross-cutting concerns that need HLD-level definition before anyone builds feature pages, plus gaps I've identified.

Shared components that must be defined before feature work splits:
(1) Auth + session shape. The scaffold already has NextAuth v4 with credentials/JWT. But your planning doc assumes role-based routing (student vs recruiter vs admin land on different dashboards) and tier-based access (bronze/silver/gold/platinum). The existing scaffold session exposes roleKeys and membershipTierRank but these don't map 1:1 to your planning doc's tier model. You need a definitive mapping: which existing Role and MembershipTier values correspond to your student/recruiter-silver/recruiter-gold/recruiter-platinum/admin tiers. This is a decision, not code - but it gates everything Sadhana builds on the login page, because post-login redirect logic depends on it.
(2) Consent service contract. This is the single most cross-cutting concern. Recruiter search (FR-R-3), admin recommendations (FR-A-7a), the recommendation gateway, and even basic student profile visibility all gate on consent status. Your planning doc defines consent as a binary toggle with per-company whitelist/blacklist, but the actual service interface needs specifying: getConsentStatus(studentId), isVisibleToCompany(studentId, companyId), getConsentedStudents(companyId). Every page that shows student data to a recruiter calls one of these. If three different devs implement their own consent check inline, you'll get inconsistent behaviour. Define the function signatures, return types, and stub implementations in src/lib/services/consent.ts before anyone builds recruiter-facing pages.
(3) Tier gate logic. Your planning doc describes TierGate as a component wrapper, which is correct for UI. But you also need a server-side equivalent for API routes - a recruiter hitting /api/students/search directly (not through UI) still needs tier validation. The scaffold's existing access-control.ts does app-level gating (userCanAccessApp) but not feature-level gating within an app. You need userCanAccessFeature(userId, featureName) that checks tier against a feature-permission map. Define the map: which features require which minimum tier. This is a config object, not complex logic, but it needs to exist centrally.
(4) Student data access layer. Multiple features query student profiles with different visibility rules: (a) recruiter search returns only consented students filtered by tier, (b) admin recommendation view returns all consented students with no tier restriction, (c) admin dashboard counts all students regardless of consent for the total count metric. If each feature writes its own Prisma query with its own WHERE clauses, consent logic will be inconsistent. Define a single getStudents(filters, callerRole, callerCompanyId) service that applies the correct visibility rules based on who's asking. Every downstream feature calls this instead of Prisma directly.
(5) Company lifecycle state machine. A company goes through: registered -> pending admin approval -> approved (with tier assignment) -> potentially suspended/banned. Your planning doc treats these as separate features (FR-A-1 approval, FR-A-2 suspend/ban) but they share state. A recruiter from a suspended company shouldn't see anything, a recruiter from a pending company shouldn't log in. This state needs a single source of truth and a single check function: getCompanyStatus(companyId) returning an enum. The suspend/ban table your doc describes (separate from User to avoid cross-app suspension) is architecturally sound but needs to be wired into the auth flow, not just the admin panel.
Gaps I've identified that aren't in your planning doc:
(1) No error handling contract. Five developers building different pages will each handle errors differently unless you define a pattern. At minimum: a shared error type, a standard API response shape ({ success: boolean, data?: T, error?: { code: string, message: string } }), and a client-side error boundary component. Without this, debugging integration issues across features will be painful.
(2) No data validation layer. Student profile fields, job posting fields, and company registration all accept user input. Your planning doc doesn't mention validation. Define a validation approach - Zod schemas co-located with your type contracts in src/types/ are the natural fit with Prisma + TypeScript. Each form submission validates against the schema before hitting the DB. Without this, you'll get inconsistent validation rules per dev.
(3) File storage abstraction. Your doc flags CV storage as TBD (PostgreSQL vs S3 vs GDrive). Regardless of where you land, define an interface now: uploadFile(file, metadata): FileRef, getFile(fileRef): Buffer, deleteFile(fileRef): void. Sadhana and whoever builds CV upload code against this interface; the actual storage backend gets swapped later. Without this abstraction, changing storage strategy later means rewriting every file-touching feature.
(4) Audit logging pattern. Your Report 1 mentions GDPR audit trail (cv_access_logs) and Report 3's schema includes it, but the planning doc doesn't define when/how logs get written. Every service that exposes student data to a recruiter should call an auditLog(action, actorId, targetStudentId, metadata) function. If this isn't defined centrally, some features will log and others won't, which defeats the GDPR compliance goal.
(5) Loading/empty/error states for shared components. Your planning doc describes stubs returning dummy data, which is correct for the service layer. But the UI components that consume these stubs also need defined states: what does a recruiter dashboard look like with zero recommended students? What does the consent toggle show during a network request? If each dev invents their own loading spinner and empty state, the UX will be inconsistent. Define a <LoadingState>, <EmptyState message="">, and <ErrorState> component set early.
How I'd structure this for Claude Code issue creation:
Split into two tiers of issues. Tier A is the architectural scaffolding (items 1-5 above plus the gaps) - these block everything and should be L1/L2. Tier B is the feature work from your planning doc table, which references Tier A services via imports. Each Tier B issue description should explicitly state which Tier A service functions it depends on, so devs know what to import.
